## Analysis - Overlay Session Data Review

Scope: review the current overlay data being displayed for practice, qualifying, and race sessions. This is semantic review only; no product changes have been made yet.

Cross-cutting findings:

- The legacy per-session overlay visibility settings are not the active gating mechanism. `OverlayAvailabilityEvaluator.IsAllowedForSession` currently allows sessions broadly, while `OverlayManager.IsAllowedForSession` only special-cases unknown-session Flags and race-only Gap To Leader. Content/header/footer session gates affect rendered rows, but some size recommendations do not shrink/grow with those gates.
- Practice and qualifying should not inherit race semantics by default. The largest repeated issue is live race-style timing or race-strategy copy appearing during non-race sessions where the underlying signal exists but means something different.
- Race behavior is generally more coherent than practice/qualifying because current models were built around race progress, pit state, standings, and fuel strategy.

Why the older three captures were updated:

- The app started as `1.0.0` at 2026-05-18 17:07:59Z, detected update `1.0.1`, downloaded/applied it, then restarted as `1.0.1` at 2026-05-18 17:08:34Z.
- On `1.0.1` startup, telemetry collection had IBT analysis and IBT telemetry logging enabled. The log says: `Startup artifact recovery found 3 raw captures missing capture synthesis or IBT analysis sidecars.`
- Recovery then wrote:
  - skipped IBT status sidecars for `capture-20260515-210810-124` and `capture-20260516-204700-385`;
  - full IBT analysis sidecars for `capture-20260518-022300-610`;
  - the new user Spa track map derived from the `capture-20260518-022300-610` IBT.
- The original capture synthesis files for the older captures were already generated near their capture end times. The May 18 changes were sidecar/recovery work, not a replay of the raw captures themselves.
- This confirms the observed behavior from testing: `1.0.0` finalized the 02:23 UTC Spa capture without IBT analysis or track-map generation, while `1.0.1` startup recovery found that older capture, matched `porsche9922cup_spa 2024 up 2026-05-17 21-23-35.ibt`, saved the IBT analysis, and generated `track-523-spa-2024-up-grand-prix-6929m-2026-02-26-04.json` with Medium confidence and 2 complete laps.
- Track-map conclusion: this was `1.0.1` startup recovery generating the Spa map, not the diagnostic bundle failing to wait long enough. The map was written at 2026-05-18 17:08:38Z; the today diagnostic bundle was created later at 17:18:03Z and its `metadata/track-maps.json` already reports the Spa user map as readable/complete. The today capture's own finalization then emitted `track_map_generation_skipped` with `reason=complete_map_already_exists`.
- Recommended behavior: keep this recovery model. Finalization should generate synthesis/IBT/map artifacts when possible; startup recovery should backfill missing sidecars after upgrades, crashes, disabled settings, or IBT write-delay cases; map generation should remain idempotent and skip when a complete map already exists. The V1.0.2 improvement is provenance: diagnostics should distinguish "generated during this capture" from "generated later by startup recovery from an older capture" and include the source capture, IBT path, generation time, confidence, and skip reason.

- Capture: 30,333 frames at 60 Hz, 0 dropped, 238 MB telemetry source, 64 session-info snapshots.

Telemetry semantics that matter for v1.0.2:

- Scoring had data in 30,111 frames, all from `SessionResults`, with `maxRows` 9. Standings should not imply full-field practice order when the official result coverage is only 9 rows out of 32 drivers.
- Gap diagnostics had data in 20,660 frames, and all of those were non-race frames. Most class gap evidence came from `CarIdxLapDistPct` inferred lap-equivalent gaps, with a smaller `CarIdxF2Time` slice. This reinforces V102-004/V102-005: non-race timing can exist but should not be presented as race gap/interval without a clear label.
- `RaceLaps` was present in every practice frame, rising from 0 to 7, but race projection produced 0 frames. Non-race race-lap signals should remain diagnostic or be carefully hidden; the current model already marks race-laps-remaining as unavailable for non-race sessions.
- Fuel level was valid in 29,841 frames and instantaneous burn in 29,271 frames. Measured burn was unavailable in all 30,333 frames because it requires a completed green-lap fuel delta. Fuel overlays should show this as level/instantaneous evidence in practice, not as a confident race plan.
- Pit-service signals existed in all frames, with 28,687 request frames and 25 service-change frames. Three pit windows were detected, none with a fuel increase. `PitSvFuel` reached 110 L selected, so Pit Service has useful practice data, but it should stay framed as service-request state rather than race strategy.
- Weather fields were present and stable: `TrackWetness` 1, `Skies` 2, `Precipitation` 0, `WeatherDeclaredWet` false. This capture does not trigger the missing-weather-field default issue, but that code path remains a risk for other sessions.
- Flags had raw active flags in all frames, but displayable flags in only 2,122 frames: black and yellow. That matches the intended "show display flags, not constant running flags" behavior.
- Track map sectors existed in all frames; highlighted sector frames appeared 963 times. Sector timing had 4 sectors, 29 crossing events, and 19 completed intervals, but also 17,249 invalid-progress frames and 22,570 lap-counter-unavailable frames. Track map/sector highlights are useful, but lap-counter-derived claims need care in practice.
- Radar had spatial rows in all frames and overlay data in 25,294 frames. No side-signal frames were recorded; 5,039 frames were unavailable due to local pit/garage context. That is appropriate for an in-car safety overlay.
- Input telemetry was strong: driver control signals existed in all 30,333 frames, with 25,974 driver-control change frames. Inputs are appropriate for practice, qualifying, and race while local context is available.

### Misses, New Signals, And Performance

Misses to act on from the v1.0.1 capture/diagnostic:

- Standings should not make a 32-driver practice session look like a complete race-style leaderboard when official scoring only produced 9 result rows.
- Standings/Relative need non-race timing treatment. The practice capture produced non-race gap data in 20,660 frames, mostly inferred from lap distance/F2 timing, so the data exists but should not be displayed as authoritative race gap/interval without clearer semantics.
- Fuel needs non-race presentation. Fuel level and instantaneous usage were strong, but measured burn was unavailable in all frames, so practice/quali should show level/usage/confidence rather than race-plan certainty.
- Race-lap signals need non-race guards. `RaceLaps` was populated through the practice capture, but race projection correctly stayed unavailable.
- Diagnostics evidence needs a rendering mode. The bundle had screenshot capture disabled and localhost had zero requests, so it cannot prove native/browser/localhost pixels.
- Wing/setup adjustment evidence is missing. The v1.0.1 capture has `WingAngle: 6` in `latest-session.yaml` and all matching session-info snapshots where setup was present, but there is no live telemetry key, IBT field, or session-info diff proving a wing change during the pit stop. That means Pit Service/setup-change rows for wing/ARB should stay out of V1.0.2 until a capture proves an actual detectable signal.

New telemetry key findings:

- No today-only live telemetry keys were captured. `capture-20260518-170923-990`, `capture-20260518-022300-610`, and `capture-20260516-204700-385` all have the same 334 live telemetry variables.
- Compared with the older NASCAR `0.20.0` capture, the Porsche/Spa live schema includes 13 fields not present there: `dcABS`, `dcDashPage`, `dcHeadlightFlash`, `dcLowFuelAccept`, `dcPitSpeedLimiterToggle`, `dcThrottleShape`, `dcToggleWindshieldWipers`, `dcTractionControl`, `dcTriggerWindshieldWipers`, `dpLFTireChange`, `dpLRTireChange`, `dpRFTireChange`, and `dpRRTireChange`.
- The older NASCAR capture had 4 fields absent from the Porsche/Spa schema: `dpLTireChange`, `dpRTireChange`, `dpWeightJackerLeft`, and `dpWeightJackerRight`. Treat these as car/session-dependent SDK shape differences, not necessarily app-version additions or removals.
- The successful IBT sidecar is the meaningful new evidence path. Today's IBT comparison found 286 IBT fields, 257 common with live, 29 IBT-only, and 77 live-only. The IBT-only fields include GPS/position (`Lat`, `Lon`, `Alt`), tire pressures/temps/speeds/ride heights (`LFpressure`, `LFtemp*`, `LFspeed`, `LFrideHeight`, and equivalents for RF/LR/RR), `BrakeABScutPct`, and `CFSRrideHeight`.
- Live-only fields remain important for overlays: the 77 live-only fields include camera/replay state, `RaceLaps`, `CarIdx*` scoring/timing arrays, `CarLeftRight`, live radio/video flags, and several `_ST` smoothed physics channels. These should stay live-overlay inputs rather than IBT/post-race-only assumptions.
- No live or IBT field matching wing, aero, ARB, setup, chassis adjustment, wedge, or weight jacker was present in the v1.0.1 Porsche/Spa schema. Setup values appeared only in session-info YAML snapshots, and those snapshots did not show the user-reported wing change.

Performance finding:

- Capture/write health looked good: 30,333 frames, 0 dropped, no write error, and no obvious GDI/USER leak in the diagnostic snapshot. Some 100 ms overlay timers averaged around 109 ms; that is worth watching in future diagnostics but is not a v1.0.2 blocker from this evidence alone.
